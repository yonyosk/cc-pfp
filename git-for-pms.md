# Git למנהלי מוצר — מה שצריך לדעת (ולא יותר)

כשעובדים עם Claude Code, נתקלים במושגים מעולם ה-git. לא צריך להיות מפתח — אבל הבנה בסיסית עוזרת לעבוד בביטחון. המדריך הזה מכסה בדיוק מה שצריך.

---

## 1. מה זה version control?

תחשבו על Google Docs. כשאתם עובדים על מסמך, Docs שומר היסטוריה של כל שינוי — ואפשר לחזור לכל גרסה קודמת.

**Git עושה את אותו הדבר, אבל לתיקייה שלמה.**

- כל קובץ בפרויקט מעוקב אחריו
- כל שינוי נשמר עם תיאור ותאריך
- אפשר תמיד לחזור לכל מצב קודם
- כמה אנשים יכולים לעבוד על אותו פרויקט בלי לדרוס אחד את השני

---

## 2. מה זה GitHub?

GitHub = האחסון בענן לפרויקטים שמנוהלים ב-git.

- כמו Google Drive, אבל ספציפית לקוד ולקבצי פרויקט
- **Public repo** — כל אחד יכול לראות ולהוריד
- **Private repo** — רק מי שהוזמן יכול לגשת

בסדנה הזו: השתמשנו ב-GitHub כדי להפיץ את ה-toolkit. מקור אחד, כולם מורידים אותו דבר — כך מבטיחים שלכולם יש אותה גרסה.

---

## 3. חמישה מושגים שבאמת צריך להכיר

### Repository (repo)
תיקיית פרויקט שמנוהלת ב-git. מכילה את כל הקבצים + את כל ההיסטוריה של השינויים.

**אנלוגיה:** תיקיית הפרויקט שלכם ב-Google Drive, אבל עם היסטוריית גרסאות מלאה.

---

### Clone
הורדת עותק של repo למחשב שלכם.

```bash
git clone [URL]
```

(או שאתם מבקשים מ-Claude Code לעשות את זה בשבילכם)

**אנלוגיה:** "תעשה לי עותק של תיקיית Google Drive הזו אל שולחן העבודה שלי."

---

### Commit
שמירת snapshot של השינויים הנוכחיים שלכם, עם תיאור קצר.

**אנלוגיה:** לתת שם לגרסה ב-Google Docs — כמו "PRD v2 - אחרי פידבק מהלקוח". לחיצה אחת שאומרת: "כאן רציתי לשמור."

---

### Branch
עותק נפרד של הפרויקט שבו אפשר לעבוד בלי להשפיע על הגרסה הראשית.

**אנלוגיה:** לפתוח duplicate של מסמך ולעבוד עליו — המקור נשאר נקי. כשמרוצים מהשינויים, ממזגים את ה-duplicate בחזרה.

---

### Pull Request (PR) / Merge
כשמסיימים לעבוד על branch, מבקשים למזג אותו בחזרה לגרסה הראשית. PR הוא הבקשה הרשמית — כולל אפשרות לסקירה לפני המיזוג.

**אנלוגיה:** שליחת מסמך לאישור לפני שמחליפים את הגרסה הרשמית.

---

## 4. מתי תשתמשו ב-git כמנהלי מוצר?

| מצב | מה עושים |
|-----|----------|
| הורדת ה-toolkit של הסדנה | `git clone [URL]` (דרך Claude Code) |
| עדכון ה-toolkit לגרסה האחרונה | `git pull` |
| שמירת שינויים ב-CLAUDE.md | `git commit` |
| ניסוי משהו חדש בלי לסכן את ה-setup | יצירת branch חדש |
| שיתוף agents ו-skills עם הצוות | push לשיתוף repo + PR |

## 6. שלושה כללי ברזל למנהלי מוצר שעובדים עם git

**לעולם אל תעבדו ישירות על `main`.**
`main` היא הגרסה הרשמית. לניסויים ולשינויים — תמיד צרו branch נפרד.

**commit קטן ותכוף עדיף על גדול ונדיר.**
כל שינוי משמעותי מקבל commit משלו עם תיאור ברור. הרבה יותר קל לחזור לנקודה ספציפית.

**תמיד pull לפני שעושים push.**
לפני שמעלים שינויים, הורידו את הגרסה האחרונה. מונע קונפליקטים.
